# CodeRabbit Live Review-to-Revision Demo — Build Brief

> **Purpose of this document:** Self-contained spec for a new Cursor agent to build an interactive demo at `/partnerships/coderabbit/demo`. Patterned on the Sentry/Datadog demos but with a different "AHA" angle — this one is about closing the **PR review loop**: when CodeRabbit (or any rigorous reviewer) leaves 14 blocking comments, Cursor addresses every one and pushes a fixup commit, all before a human re-reads the PR.

---

## 0. TL;DR for the agent

Build a repeatable, click-to-run demo that dramatizes CodeRabbit + Cursor orchestration end-to-end:

1. A user lands on `/partnerships/coderabbit/demo`. The hero is a pixel-perfect GitHub PR view for `PR #284 · feat: introduce billing-v2 service` with the **CodeRabbit review summary** at the top: "14 actionable comments · 3 critical · 7 nitpicks · 2 suggestions · 2 docs". The PR currently has 0 humans assigned and is blocked on review.
2. The user clicks **"Hand to Cursor agent"**. Briefly (~3s), the GitHub UI shows a "Cursor agent reading 14 comments…" indicator, then pivots to a full-screen takeover: a **per-comment triage table** showing how Cursor will address each comment (auto-address, push back politely, or escalate to author).
3. The user clicks **"Watch Cursor address the review"**. Split screen: the live PR conversation thread on the left (with new comment rows + diff updates animating in), agent console on the right.
4. The agent: reads each comment in order of severity, applies the requested change for the 9 it can confidently auto-address, posts a **respectful structured pushback** for the 3 it disagrees with (with cited reasoning), and asks 2 clarifying questions for the genuinely ambiguous ones. It pushes a single squashed `fixup!` commit, marks the addressed threads as resolved, and re-requests review.
5. When complete, four artifact modals are clickable: **Updated PR (with all 14 threads in their final state)**, **Fixup diff**, **Pushback rationale doc**, and a **Reviewer-time comparison card**.
6. Reset returns to the pre-fixup state. `scripts/reset-coderabbit-demo.sh` re-seeds the original review.

**The "AHA":** *"Cursor handled my whole CodeRabbit review while I was in standup."* Senior engineers immediately understand: review-loop latency is the single biggest hidden tax on velocity, and this is the demo that makes it disappear without sacrificing rigor — Cursor pushes back when it should, asks when it should, fixes when it should.

---

## 1. Why this demo, why this angle

CodeRabbit's superpower is exhaustive AI code review. Its weakness — and every reviewer's weakness — is that the human author still has to address every comment. That's the bottleneck. The most impactful agentic story for CodeRabbit isn't "Cursor catches bugs CodeRabbit missed"; it's "Cursor closes the review loop CodeRabbit opens." This positions Cursor as the **executor** complementing CodeRabbit's **reviewer** role. Mutually reinforcing, not competitive.

Two specific differentiators to lean into:

- **Calibrated disagreement.** The demo must show Cursor *politely refusing* a CodeRabbit suggestion when the suggestion is wrong (with cited reasoning). This is the most credible thing the demo can do.
- **Conversation-shaped artifacts.** Unlike the Sentry/Datadog demos where the artifact is a single-page result, this demo's artifact is a *conversation thread*. Treat that thread as the centerpiece.

## 3. The demo — concept & story

### Trigger UI (idle phase)

A pixel-perfect **GitHub PR view** for `PR #284 · feat: introduce billing-v2 service`. Components:

- Top: standard GH PR header with status (`Open · Awaiting review`), branch (`feat/billing-v2 → main`), assignees (none), reviewers (only `coderabbitai[bot]`).
- Below: the **CodeRabbit review summary card** — pinned at the top of the conversation. Includes:
  - The CodeRabbit walkthrough panel ("This PR introduces a billing v2 service with…")
  - A summary table: `Severity · Count` with critical/nitpick/suggestion/docs.
  - A "Files reviewed" breakdown.
  - A small CodeRabbit logo and "Generated by CodeRabbit · gpt-5.4 · 2.4s" footer.
- Below: a "Conversation" tab and "Files changed (12)" tab. Default to Conversation. Scroll-down should reveal 4–5 of the 14 inline comments as a teaser (each rendered as a real GH inline-comment card with file/line annotation, reviewer avatar, comment body, and a "Resolve" button).
- Bottom-right floating action: a single CTA `[ Hand to Cursor agent ]` styled as an extension of the GH actions panel.

Below the fold:

- A **Reviewer-time comparison card** (the analog of `cost-comparison.tsx`):
  - Without Cursor: dev reads 14 comments, addresses 9, debates 3, asks PM 2 → **3.5h dev time**, plus 1 reviewer re-read (1h) → **4.5h round-trip**.
  - With Cursor agent: agent addresses + responds + asks → **6 min**, then 1 reviewer re-read (1h) → **1.1h round-trip**.
  - Subhead: 4× faster review loop, dev shifts from clerical-comment-handling to next feature.
- A **Guardrails panel** specific to CodeRabbit/code-review:
  - "Auto-addresses comments only when the requested change is local and the existing test suite covers the touched path."
  - "Pushes back with cited reasoning when CodeRabbit's suggestion conflicts with project conventions."
  - "Escalates to the human author when intent is ambiguous."
  - "Never marks a thread resolved without making the requested change OR posting a justified disagreement."
  - "Force-push disabled. Adds `fixup!` commits only."

### The triage table (planning takeover)

Click **Hand to Cursor agent**. ~3s "reading 14 comments…" indicator. Pivot to a full-screen **comment-triage takeover**:

- Top: PR title and CodeRabbit review summary.
- Body: a 14-row table, one row per comment. Columns:
  - `#` (1–14)
  - Severity chip (`critical`, `nitpick`, `suggestion`, `docs`)
  - File:line
  - One-line summary of the comment ("`amountInCents` should be a `Decimal`, not a `number`")
  - **Cursor's planned action** (chip):
    - `Auto-address` (9 rows, blue)
    - `Push back · cite policy` (3 rows, amber)
    - `Ask author · clarify intent` (2 rows, purple)
  - Confidence (0.00–1.00)
- Bottom: `[ Watch Cursor address the review ]` and `[ Reset ]`.

### The orchestration (running phase)

Split screen:

- **Left:** the **live PR conversation thread**. As the agent works, comments are addressed in order. Each addressed comment animates a state change:
  - Auto-address → comment thread shows a new reply from `cursor-agent` ("Done in <commit-sha>. Thread resolved."), then the thread visually collapses with a green "Resolved" pill.
  - Push back → `cursor-agent` reply with structured rationale (citing the project convention or a counter-example), comment stays open with an "Author response requested" pill.
  - Ask author → `cursor-agent` reply with a clarifying question, status badge changes to "Awaiting author".
  - The diff view (toggle at the top of the panel) updates live to reflect the fixup commit's contents.
- **Right:** scripted agent console. Channels:

| Channel       | Label                  | Hex accent | Role                                                  |
| ------------- | ---------------------- | ---------- | ----------------------------------------------------- |
| `coderabbit`  | `coderabbit-mcp`       | `#FF5722`  | Read review summary, comment metadata, rule citations |
| `github`      | `github-mcp`           | (white)    | Read code, post replies, push fixup commit            |
| `composer`    | `composer · address`   | blue       | Per-comment scoped patches                            |
| `opus`        | `opus · disagree`      | `#D97757`  | Drafts the structured pushback responses              |
| `codex`       | `codex · review`       | `#10a37f`  | Quick re-review of the fixup before push              |
| `shell`       | `shell`                | green      | tsc, lint, focused unit tests                         |
| `done`        | `complete`             | green      | Terminal step                                         |

### Target script arc (~30 steps, ~22s real, scaled to ~6m displayed)

1. **Intake (coderabbit + github):** Fetch PR, fetch CodeRabbit review, fetch diff, fetch commit history.
2. **Plan (opus):** Per-comment triage. Same 9/3/2 split shown in the planning page.
3. **Auto-address loop (composer + shell):** For each of 9 auto-address comments:
   - Read the file at the cited line.
   - Apply the change.
   - Post a reply on the thread linking to the fixup-commit SHA.
   - Mark thread resolved.
   The console shows these as a tight burst of activity. Group some by file (`amount.ts` gets 3 changes in one composer call to avoid choppy script).
4. **Disagree loop (opus + github):** For each of 3 pushback comments:
   - Opus drafts a respectful, structured response. Format must always be: *"Thanks for catching this. After review I'd push back here because <cited reason>. Happy to apply if you still prefer it."* Cite a project convention file (`docs/eng-conventions/decimals.md`) or a code reference (`src/utils/money.ts:14`).
   - GitHub posts the reply, leaves the thread open.
5. **Ask loop (opus + github):** For 2 ambiguous comments, post a one-line clarifying question.
6. **Static verify (shell):** Run `tsc --noEmit`, `eslint`, and the focused unit tests for the touched files. ~12s of real wait elsewhere; here scale appropriately. Show pass/fail rows.
7. **Codex review (codex):** Quick scan of the fixup diff against the original PR for unintended side effects.
8. **Push fixup (github):** A single squashed `fixup!` commit pushed. Force-push disabled — explicit row showing "skipped force-push, used new commit".
9. **Re-request review (github):** Mark `coderabbitai[bot]` as re-request reviewer.
10. **Done.**

### The four artifact modals

All MacBook-framed.

1. **Updated PR (`updated-pr-modal.tsx`)** — pixel-perfect updated GitHub PR view. The conversation tab is the centerpiece:
   - 9 collapsed "Resolved" threads at the top.
   - 3 "Awaiting author response" threads in the middle (the pushbacks) — each fully visible with the cited rationale.
   - 2 "Awaiting author" threads at the bottom (the clarifying questions).
   - One new commit row at the bottom of the timeline: `cursor-agent · fixup! address coderabbit review (12 changes across 7 files)`.
   - CI status: all green.
   - "Re-requested review from coderabbitai[bot]" event.
   This is the demo's most important artifact — invest in pixel parity with GitHub's actual conversation chrome (avatars, "marked as resolved" affordances, "show resolved (9)" toggle).

2. **Fixup diff (`fixup-diff.tsx`)** — split-pane diff viewer for the fixup commit. Show 7 files changed with realistic per-file hunks. Color-code each hunk's left margin to match the comment severity that drove it (blue auto-address, amber pushback-then-applied, etc.) — a small but powerful touch.

3. **Pushback rationale (`pushback-doc.tsx`)** — a single markdown document collecting the 3 pushback responses with full cited reasoning. Each section is structured:
   - The original CodeRabbit comment (block-quoted).
   - The cited project convention or counter-example (with file:line link).
   - Cursor's recommended action and rationale.
   - Suggested follow-up if the reviewer still disagrees.
   This artifact is the most powerful credibility lever in the demo. Make it read like a thoughtful senior engineer wrote it, not a chatbot.

4. **Reviewer-time card (`time-saved-card.tsx`)** — same pattern as the cost card but in modal form, with a single big number ("4.1× faster review loop · 3.4h dev time recovered per PR") and a small footnote.

### Branding

- **Primary accent:** CodeRabbit orange `#FF5722` for the CodeRabbit logo + review-summary chip + "rules cited" badges. Use sparingly.
- **GitHub-native:** the entire idle/running/complete experience should look like GitHub. Match GH's conversation chrome closely (deep-mode `#0d1117` backgrounds, GH avatars, the resolve-thread affordance, etc.).
- **Pushback amber, ask purple, auto-address blue, resolved green** — keep these chip colors consistent across the planning table, the live thread, and the artifact modals. This is the demo's color grammar.
- **Vocabulary:** "comment", "thread", "resolve", "fixup commit", "re-request review", "actionable comments". Avoid "ticket", "issue", "incident".

## 7. Anti-patterns

- ❌ **Don't make Cursor agree with every CodeRabbit comment.** The pushback responses are the most credible part of the demo. Make them substantive, citing project conventions or counter-examples. A demo where Cursor obediently follows every reviewer suggestion is a demo nobody buys.
- ❌ **Don't use the existing `pr-modal.tsx` from Sentry/Datadog as the artifact.** Those PRs are unreviewed; this PR is **post-review with conversation as the centerpiece**. Build a new PR component that prioritizes the conversation tab over the diff tab.
- ❌ **Don't show Cursor force-pushing.** Always a `fixup!` commit. The guardrails panel says so; the demo should match.
- ❌ **Don't skip the "ask" bucket.** A demo where Cursor only addresses or pushes back misses the third move (asking). Calibrated agents *ask*. Show that.
- ❌ **Don't put CodeRabbit copy that sounds like Cursor wrote it.** Read a few real CodeRabbit reviews and match the voice — slightly more formal, very rule-citation-heavy, includes diff suggestions inline.

## 8. Webhook prompt (`buildAgentPrompt`)

Fork the Sentry/Datadog webhook shape. Required steps in order:

1. **Intake.** Fetch the PR, the full CodeRabbit review, the diff, the commit history, the relevant project convention docs.
2. **Per-comment triage.** Bucket each comment into `auto-address`, `push back`, or `ask`. The thresholds:
   - Auto-address only if: the requested change is local, the existing tests cover the touched path, and the change is consistent with project conventions.
   - Push back if: a project convention contradicts the suggestion, OR a counter-example exists in the codebase, OR the suggestion would introduce a regression.
   - Ask if: intent is genuinely ambiguous and a reasonable engineer would also need to ask.
3. **Execute auto-address.** Apply the patches, run focused tests for the touched paths, post a reply linking the fixup-commit SHA, mark thread resolved.
4. **Execute pushback.** Draft a respectful, structured response citing the project convention or counter-example. Post but leave thread open.
5. **Execute ask.** Post a single clarifying question.
6. **Static verify.** `tsc`, `lint`, focused unit tests. Iterate until clean.
7. **Push.** Single squashed `fixup!` commit. **Never force-push.**
8. **Re-request review.**
9. **PR comment summary.** Top-level comment: "Addressed 9, pushed back on 3 (rationale linked), asked 2 clarifying questions. Fixup commit: `<sha>`."

The prompt must enforce: **never resolve a thread without making the requested change or posting a justified disagreement**, **never force-push**, **always cite the source of any disagreement**.
